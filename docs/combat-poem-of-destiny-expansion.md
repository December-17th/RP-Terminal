# 命定之诗 Combat Expansion — Design (in progress)

Status: **Design in progress (2026-06-25).** The RP Terminal **app** combat system is fully built &
committed (branch `feat/combat-system`, see [combat-system-design.md](combat-system-design.md) +
[plans/2026-06-25-combat-system.md](superpowers/plans/2026-06-25-combat-system.md)). This doc is the
**next phase: a combat expansion for the 命定之诗 CARD itself** — NOT app changes. The app's native
combat engine (`src/shared/combat`, `combatService`) and the `extensions.rp_terminal.combat` bundle
slot are the **fixed contract** we author against.

## Scope (owner-locked via Q&A, 2026-06-25)
- We are developing an **expansion for the 命定之诗 character card**, not modifying the RP Terminal app.
- Use the **card's own custom schema** (5 attributes, not D&D 6). Hybrid: **map what exists, define the
  rest**. Equipment is **fully slotted**.
- Combat numbers (damage dice / to-hit / AC) are **generated by the AI via `<UpdateVariable>`** when the
  AI creates the parent item/ability, and **live in the MVU variable system** (on each item). The
  **MVU UI displays them** (same variable system — no separate bundle catalog). Owner is "leaning toward
  modifying the MVU UI to display them and use the same variable system."
- The player must be able to **view abilities + their combat details** in the MVU UI.
- Card-UI edits are delivered as a **standalone regex JSON** to re-import (the status UI is currently a
  remote loader; see below).

## The 命定之诗 stat_data schema (recovered — authoritative)
Fetched from `FrontEnd-for-destined-journey@1.8.2/dist/data_schema/index.js`. Top-level: `事件`, `世界`
`{时间,地点}`, `任务列表`, **`主角`** (the player), `命运点数`, **`关系列表`** (companions, keyed by name —
same character fields + `在场`/`好感度`/`性格`/…), `新闻`.

A **character** (`主角` and each `关系列表[name]`):
- `种族`, `身份[]`, `职业[]`, `生命层级`, `等级`(1–25), `累计经验值`, `升级所需经验`, `冒险者等级`, `属性点`
- **`属性`**: `{ 力量, 敏捷, 体质, 智力, 精神 }` (5 numeric attributes)
- **`生命值上限` / `生命值`**, `法力值上限`/`法力值`, `体力值上限`/`体力值` (flat number pairs, NOT {value,max})
- **`状态效果`**: record<key, `{类型(增益/减益/特殊), 效果, 层数, 剩余时间, 来源}`> (buffs/debuffs)
- `金钱`
- **`背包`**: record<item, `{品质, 类型, 数量, 标签[], 效果:record<string,string>, 描述}`>
- **`装备`**: record<slot, `{品质, 类型, 标签[], 效果:record<string,string>, 描述, 位置}`>
- **`技能`**: record<name, `{品质, 类型, 消耗, 标签[], 效果:record<string,string>, 描述}`>
- `登神长阶` (ascension): `{是否开启, 要素, 权能, 法则, 神位, 神国{名称,描述}}`

**The crux:** there are **NO machine-readable combat fields** — no damage dice, to-hit, AC,
attack/defense. `效果` is a **string→string record (narrative prose)**. So the expansion must *map* the
attributes/HP/items that exist and *define* the combat numbers the engine needs.

## The locked model — combat fields are AI-authored MVU variables on each item
Add a formalized **`战斗` sub-object** to weapons/skills/abilities (keeps narrative `效果` separate from
mechanics). The AI emits it via `<UpdateVariable>` when it creates the parent item/ability.

```jsonc
// on a 技能 / weapon (装备 whose 类型 is a weapon):
"战斗": {
  "伤害": "2d6+智力",                       // dice; ability tokens = the 5 attrs
  "命中": "力量",                           // to-hit attribute; null ⇒ save-based
  "豁免": { "属性": "敏捷", "难度": 14 },     // optional, save-based
  "射程": 6,                                // cells
  "范围": { "类型": "爆发", "半径": 2 },      // 自身 / 爆发 / 直线 / 锥形
  "消耗槽": "攻击",                          // "攻击" | "动作"  (action economy slot)
  "需视线": true,
  "命中后": [{ "状态": "灼烧", "回合": 2 }]
}
// on armour:  "战斗": { "护甲": 2 }
// on a 被动:  "战斗": { "修正": { "命中": 1 } }
```

The `rp_terminal.combat` bundle then shrinks to:
- **`stat_map`** — `player:"主角"`, party from `关系列表` (those with `在场:true`); field paths for
  `属性`, `生命值`/`上限`, `等级`, `装备`, `技能`, `状态效果`.
- **`derive`** — attribute→modifier scale (`mod = floor((attr − base)/div)`, bundle-configurable for the
  card's scale), `AC = 10 + 敏捷mod + Σ equipped 护甲`, default speed.

A **new** `buildEncounterFromMvu(statData, stat_map, derive)` (sibling to the template-based
`buildEncounter` in `src/shared/combat/bundle.ts`) reads each party member: attributes→mods, HP from
`生命值`/`上限`, AC derived, equipped weapon `战斗`→a default attack, `技能` with `战斗`→abilities,
`被动`→modifiers, `状态效果`→conditions. Missing `战斗` → safe fallback (unarmed `1d4`). Fold-out writes
HP/resource deltas back to these paths via MVU.

## MVU UI
Extend the card's character/status UI to render the combat sheet: 5 attributes (+ derived mods),
HP/法力/体力, derived AC, the equipped weapon's attack profile, **skills-as-abilities with combat
details (damage/range/cost/effects)**, passives, status effects. Delivered as a **self-contained
status-area regex JSON** (reads `stat_data`, computes derived numbers with the same `derive` formulas).

**Card UI inventory (verified from `v4.2.1.png` `regex_scripts`):**
- `角色查看器v3.0.5` — 64 KB **inline Vue** character sheet (reads `stat_data` via `getVariables`).
- `状态栏` — **remote** loader → `FrontEnd-for-destined-journey@1.8.2/dist/status/index.html` (built/minified; can't hand-edit → we ship a standalone regex instead).
- `战斗&生产制作美化` — 28 KB inline "TRPG战斗面板" **beautification** of AI-written combat text →
  **the OLD AI-narration combat paradigm, SUPERSEDED by the engine's grid CombatView** (retire/leave it).

2. **Attribute → modifier scale** — need one real character's `属性` values (typical starting `力量`?) to
   set the `derive` defaults. **Fetch an example / the status bundle, or ask the owner.**
3. **AC** — derive (`10 + 敏捷mod + 护甲`) [leaning] vs AI-emitted total.
4. **Field names** — confirm `战斗 / 伤害 / 命中 / 射程 / 范围 / 消耗槽 / 需视线 / 护甲` fit the card's voice.
